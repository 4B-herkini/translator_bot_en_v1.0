\## 2. 서비스 핵심 요약



\### 문제 정의

\- 한국어는 비정형 입력 (구어체, 어순 붕괴)이 많음

\- 기존 번역기는 “정제된 입력”을 전제로 설계됨



\### 솔루션

1\. 한국어 정제 (전처리)

2\. 다중 톤 번역 (Professional / Friendly / Concise)



👉 핵심 차별:

\*\*입력 정제 → 번역 2단 구조\*\*



\---



\## 3. 강점 분석



\### 3.1 문제 정의 정확도 (매우 높음)

\- 실제 사용자 pain을 정확히 찌름

\- 특히 음성 입력 환경에서 효과 큼



👉 상위 10% 수준 문제 정의



\---



\### 3.2 UX 방향성 우수

\- “아무렇게나 입력해도 된다”는 가치 명확

\- 입력 부담 제거 → 사용성 상승



\---



\### 3.3 타겟 시장 적절

\- 직장인 / 사업자 / 개발자

\- 반복 사용 + 지불 의사 있음



👉 SaaS 가능성 있음



\---



\### 3.4 가격 구조 현실적

\- 9,900원 / 29,900원

\- 국내 SaaS 시장 기준 적절



\---



\## 4. 핵심 문제 (사업 리스크)



\### 4.1 기능 대체 가능성 (가장 치명적)



현재 서비스는 사실상:



👉 “프롬프트 자동화 레이어”



사용자는 이미 다음과 같이 해결 가능:

\- 정제 + 번역 + 톤 지정 → AI 직접 요청



👉 결론:

\*\*핵심 기능이 독점적이지 않음\*\*



\---



\### 4.2 해자(Moat) 부재



현재 없음:

\- 데이터 락인

\- 네트워크 효과

\- 독점 기술



👉 UI/UX 차별화 수준 → 복제 쉬움



\---



\### 4.3 대기업 대응 취약



가능 시나리오:

\- AI 서비스에 “정제 + 톤 번역 버튼” 추가



👉 서비스 존재 이유 약화



\---



\### 4.4 사용자 행동 미스매치



현실:

\- 사람들은 별도 번역 사이트를 열지 않음

\- 기존 툴 안에서 해결하려 함



👉 웹앱 단독 구조는 불리



\---



\### 4.5 비용 구조 리스크



\- API 비용 기반 구조

\- 사용량 증가 = 비용 증가



👉 헤비 유저 많아질수록 마진 압박



\---



\## 5. 전략적 개선 방향



\---



\### 5.1 “번역” → “업무 결과물”로 확장



기존:

\- 번역 제공



개선:

\- 실제 사용할 문장 생성



예:

\- 이메일

\- 클레임 대응

\- 견적 전달



👉 “복붙 가능” 수준으로 업그레이드



\---



\### 5.2 독립 서비스 → 붙는 서비스



필수 방향:

\- 크롬 확장

\- 슬랙 연동

\- 지메일 플러그인



👉 기존 업무 흐름 안으로 들어가야 함



\---



\### 5.3 도메인 특화



범용 → 실패 확률 높음



추천:

\- 무역

\- IT 커뮤니케이션

\- 해외 셀러



👉 용어 + 히스토리 = 해자 생성



\---



\### 5.4 데이터 축적 구조 구축



필수 요소:

\- 사용자 문장 히스토리

\- 회사별 톤

\- 반복 패턴



👉 락인 효과 발생



\---



\### 5.5 AI 확장



\- 자동 문장 생성

\- 상황별 템플릿

\- 추천 시스템



👉 “번역기 → 커뮤니케이션 엔진” 전환
